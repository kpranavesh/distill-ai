04 · Article output anatomy — what the user actually sees per article
comfortSummary The article body
Source
// route.ts:221
buildComfortSummary({
  comfort,
  base: article.summary || article.title
})
What it does
Takes raw RSS snippet verbatim, appends ONE comfort-tuned sentence.
Comfort suffixes
skeptic: "Think of this less as hype..."
beginner: "If any jargon shows up..."
active: "The question for you is..."
power: "Read this like a changelog..."
Problem
Length is unbounded — RSS feeds vary from 3 words to 3 paragraphs. No truncation applied.
whyItMatters The LLM sentence
Source
Groq Llama 3.1 8B
model: llama-3.1-8b-instant
max_tokens: 1024
all 8 articles in 1 prompt
Prompt constraint
"write exactly ONE sentence
(under 22 words)..."
Context sent
role + industry
+ goal + comfort level
+ title + summary[:160]
Fallback (Groq fails)
Rule-based by role.
_debug.whySource = "fallback"
Quality
Tightest, highest signal text the user sees. Role + industry personalized. 22-word hard cap.
tryThis The action prompt
Type definition
// page.tsx:35
tryThis?: string
API response
// route.ts:213–223
items = ranked.map(a => ({
  ...
  comfortSummary,
  whyItMatters
  // tryThis not included
}))
Status
Field exists in type and static examples but is never populated from API. Dead field.
Opportunity
Add to Groq prompt — return whyItMatters + tryThis as JSON object per article.
05 · Areas of opportunity — prioritized
Fixed ✓
goals[0] only — multi-select data thrown away
Was: API received only the first goal. All other selections were silently ignored.
// Fixed: for (const goal of profile.goals) // Each goal now scored independently in score.ts
Resolved: page.tsx sends goals=build,find-tools. route.ts parses to Goal[]. score.ts loops over all goals — each keyword group contributes independently. stay-informed/curiosity share a single +8 applied once via .some().
High · Waste
Name question burns onboarding time for zero gain
Step 1 asks for a name. It is only used to personalise the greeting string. No effect on ranking, scoring, or personalization of summaries.
// generateDistillReply — name unused // whyItMatters prompt — name unused // scoreArticle() — name unused
Fix: Remove from onboarding. Ask post-signup in a profile settings page, or drop entirely. Use the freed space to ask something that scores (seniority, company size).
High · Quality
comfortSummary = raw RSS with an appended sentence
The article body shown to users is an unprocessed RSS snippet that can be 3 words or 500 words. Only one comfort-tuned sentence is appended — the core isn't personalized at all.
// route.ts:142 — base = article.summary verbatim // no truncation, no cleaning, no rewrite
Fix: Truncate base to 120–150 chars before appending. Or include comfortSummary in the Groq prompt as a rewrite pass — costs ~2 extra tokens per article.
High · Coverage
Industry scoring fires weakly for non-tech verticals
All 13 RSS feeds are AI/tech-focused. Healthcare, finance, legal, retail users get keyword matches only when AI news happens to mention their sector. The tech blanket +6 bonus has no equivalent for others.
// CLAUDE.md: "Industry-specific scoring fires // weakly when feeds don't contain sector keywords"
Fix: Add 1–2 sector-specific feeds per vertical (e.g. STAT News for healthcare, Financial Times AI for finance). FEEDS array is the only change needed.
Medium · Signal
Technicality is binary — one keyword = same as twenty
isTechnical is a boolean. An article with one technical term receives identical treatment as a deeply technical research paper. This creates false negatives for power users and false positives for skeptics.
// score.ts:62 — boolean, not a score const isTechnical = contains(corpus, TECHNICAL_KEYWORDS) // opportunity: countMatches → 0 / 1 / 2+ tiers
Fix: Replace with countMatches — score 0 (none), 1 (light), 2+ (deep). Apply graded bonuses/penalties. Minimal code change, better signal quality.
Medium · Dead field
tryThis field defined but never populated
BriefingItem type includes tryThis?: string and it appears in static examples with high quality copy. But route.ts never returns it — the field is always undefined in live briefings.
// route.ts:213–223 — tryThis absent from .map() // page.tsx:35 — tryThis?: string (typed, never set)
Fix: Extend Groq prompt to return {"why": "...", "try": "..."} per article. One structural change to the prompt — no new API call.
Low · Signal
No seniority or company size signal
An IC engineer and a VP of Engineering get identical role scoring. A 5-person startup founder and a Fortune 500 exec get the same treatment. These are high-signal inputs that could sharpen both content ranking and the whyItMatters prompt.
// No seniority or company size in UserProfile // recommender/types.ts — role: string only
Fix: Add optional seniority (individual / manager / director / exec) and optionally org size. Use in both scoring and whyItMatters prompt. Could replace the name question slot in onboarding.
Low · UX
Goals UI misleads — multi-select implies multi-use
Multi-select implies all selected goals matter. Users may select 3 goals assuming they all influence results. Currently only goals[0] matters, which means the selection order at click time determines the outcome.
Fix (minimal): Change to radio / single-select to match current model behavior. Fix (better): Implement multi-goal scoring (see critical bug above) so multi-select is actually honoured.
Low · Insight
Add "what are you trying to decide right now?"
A one-line free text input at onboarding (or in chat) describing an active problem or decision would dramatically sharpen whyItMatters personalization. It's the highest-signal single input possible — but requires passing it into the Groq prompt.
Fix: Add optional "active context" field to UserProfile. Include in whyItMatters prompt as "user is currently trying to: [context]". No scoring change needed — LLM personalization only.
