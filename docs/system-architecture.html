Caching layer
Three independent caches, all must be defeated
Next.js patches fetch and caches responses by default — unstable_noStore() opts out at the function level. The Vercel edge CDN caches before the serverless function runs — CDN-Cache-Control: no-store stops it. The browser caches identical GET URLs — _t: Date.now() makes every request URL unique. All three are active in this codebase.
RSS Layer
13 feeds, parallel fetch, 120-article pool
All feeds are fetched concurrently via Promise.all() with an 8-second per-feed timeout. Articles are flattened, sorted by published descending, then capped at 120 before scoring. The bottleneck is always the slowest RSS feed — scoring 120 articles is near-instant by comparison. The 120 cap keeps scoring time negligible as feed count grows.
LLM Layer
One Groq call for all 8 articles, rule-based fallback
All 8 ranked articles are sent in a single prompt to Llama 3.1 8B (Groq free tier). The model returns a JSON array of 8 personalized sentences, each under 22 words, referencing the actual article content. If Groq fails or returns malformed output, a role-aware rule-based fallback fires silently. _debug.whySource tells you which path ran — "claude" (Groq succeeded) or "fallback".

04 Narrative — design decisions and how each piece connects
Onboarding → Scoring
Six questions, five scoring inputs, one wasted slot
The onboarding collects six inputs across three steps. Name is cosmetic — it goes into a greeting string and nothing else. The remaining five — role, industry, comfort, goals, and AI tools — all feed directly into scoreArticle().

Role and industry are the context inputs: they tell the engine what domain and job function matters. Comfort level is the depth filter: it controls whether technical content is a boost or a penalty, and it's the single factor with the widest swing (±30 pts between power and skeptic). Goals are the intent filter: they select which keyword groups get scored. AI tools are a negative signal only — they suppress intro content for tools the user already knows.
Multi-goal scoring (updated)
Each goal now contributes independently to every article
Previously, the scoring engine ran a switch(profile.goal) on a single string — only goals[0] was ever evaluated. If a user picked "Find tools" and "Build things", the build intent was silently ignored.

The fix replaces the switch with a for...of loop over profile.goals. Each goal hits its own keyword group — TOOL_LAUNCH_KEYWORDS, STRATEGIC_KEYWORDS, or BUILD_KEYWORDS — and adds its contribution independently. stay-informed and curiosity share the same flat +8 news signal, applied once via .some() after the loop to prevent double-counting.

The per-goal cap remains at +24. Because multiple goals can stack, the theoretical raw maximum rises to 132, but the final clamp(0, 100) ensures no article ever exceeds 100. Scores only inflate when an article genuinely hits multiple keyword groups — which is a signal that it's truly broad-interest content.
Scoring model design
Keyword-based, not ML — fast and fully inspectable
Every factor in scoreArticle() is a keyword lookup or a rule. There's no model to train, no embeddings, no semantic similarity. This is intentional — the entire scoring logic is readable in a single file, every decision is auditable in the scoreBreakdown[] array, and it runs in microseconds across 120 articles.

The tradeoff is coverage: an article about "AI in emergency rooms" won't score highly for a healthcare user unless the article happens to use specific sector terms. The real fix there is feed diversity, not model complexity.
Article text pipeline
Two text fields: RSS verbatim + LLM one-liner
Each article surfaces two text outputs. comfortSummary is the raw RSS contentSnippet with one comfort-tuned sentence appended — it's uncontrolled in length (can be 10 words or 500). whyItMatters is the tight output: Groq returns one sentence per article, under 22 words, written specifically for the user's role and industry.

If Groq fails, a role-aware fallback generates a generic question prompt per article. _debug.whySource in the API response tells you which path ran.
API contract
Query params in, JSON out — no request body needed
The briefing API is a plain GET endpoint. All user context arrives as query params: role, industry, comfort, goals (comma-separated), aiTools (comma-separated), and _t (timestamp for cache-busting). The response is a JSON object with items[] and a _debug block.

No auth token is passed in the query string — authentication is via Supabase session cookie, verified server-side at the top of the handler. If the session is invalid, the handler returns the auth error before touching RSS feeds.
What's still on the floor
Known gaps in the current model
Technicality is binary. isTechnical is a boolean — one matching keyword has the same weight as twenty. An article barely mentioning "api" penalises a skeptic as hard as a deep benchmark paper. Grading this 0/1/2+ tiers would improve signal quality with minimal code change.

Industry coverage is skewed. All 13 RSS feeds are AI/tech-focused. Healthcare, finance, legal, and retail users score on industry keywords only when AI news happens to mention their sector — which is rare. The tech industry gets a blanket +6 that no other vertical has. Adding 1–2 sector-specific feeds per vertical is the fix.

tryThis is a dead field. The BriefingItem type declares tryThis?: string and the static onboarding examples show it with high-quality copy. But the live API never populates it — the field is always undefined. Extending the Groq prompt to return both whyItMatters and tryThis as a JSON object would activate it with no new API call.

Seniority and company size are missing. An IC engineer and a VP of Engineering score identically. A 5-person startup founder and a Fortune 500 exec get the same treatment. Both are high-signal differentiators for content ranking and for personalizing the Groq prompt.
